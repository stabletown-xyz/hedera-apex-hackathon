# Stabletown API — v1

All endpoints are tenant-scoped and authenticated.

## Global conventions

### Contract source of truth

OpenAPI artifact:

* `contracts/openapi/openapi.v1.json` (generated)

Generated type artifacts:

* `frontend/operator-console/src/types/generated-api.ts`
* `sdk/packages/sdk/src/generated/openapi-types.ts`

Verification commands:

* `mix stabletown.openapi.verify`
* `mix stabletown.types.verify`

### Base URL

* `/api/v1`

### Auth

* `Authorization: Bearer <jwt>`

### Tenant scoping

One of:

* `X-Tenant-Slug: <tenant_slug>` header, or
* `X-Tenant-Id: <tenant_id>` header (current scaffold implementation path), or
* subdomain routing (e.g., `https://<tenant>.api...`)

**Rule:** every request must resolve `tenant_id` and all queries must filter by it.

### Idempotency (required for money-adjacent writes)

Money-adjacent endpoints must accept:

* `Idempotency-Key: <string>` header (recommended)

**Idempotent replay rule:** if a request with the same idempotency key is repeated, return the same semantic outcome without double-posting to the ledger.

### Feature-flagged additive money-surface metadata

When tenant flag `cari_dvp_policy_metadata_enabled` is enabled, DvP-annotated artifacts may include additive metadata references:

* `money_form_ref`:
  * `form_id`
  * `kind` (`pay_balance|park_balance|deposit_token|payment_stablecoin|entitlement|security_position|collateral_position|escrowed_funds`)
  * `form_ref`
  * `settlement_rail` (nullable)
* `settlement_intent_ref`:
  * `intent_ref`
  * `source_type`
  * `source_id`
  * `requested_rail` (nullable)
  * `selected_rail` (nullable)

These fields are additive and do not alter endpoint names, idempotency semantics, or baseline response compatibility.

### Time

* Timestamps are ISO8601 UTC.

### Money amounts

* Integers in cents: `amount_cents`
* Currency code string: `currency`

### Standard response envelope (recommended)

For consistency, prefer:

```json
{
  "status": "ok",
  "data": {}
}
```

### Errors

Use stable error codes:

* `unauthorized`
* `forbidden`
* `not_found`
* `validation_error`
* `insufficient_balance`
* `idempotency_conflict`
* `internal_error`

Example:

```json
{
  "status": "error",
  "error": {
    "code": "validation_error",
    "message": "name is required",
    "details": {"field": "name"}
  }
}
```

## Hedera Live Cutover Notes (Munus)

The Hedera live cutover does **not** rename or replace Munus routes.

- Namespace remains `/api/v1/munus/*`.
- Deploy/open/close/control API contracts are unchanged; runtime execution switched from deterministic simulation to a live CLI-backed adapter path.
- Hedera runtime adapter envelope remains stable (`ok`, `tx_hash`, `status`, `failure_reason`, `metadata`).

### POST /merchants/:id/offline-payments/sync

Record and synchronize one offline terminal payment into canonical redemption flows.

**Headers**

* `X-Tenant-Id: <tenant_id>` (required)
* `Idempotency-Key: <string>` (required; money-adjacent write)
* `X-Actor-Id: <uuid>` (optional)
* `X-Actor-Role: <role>` (optional; write roles: `tenant_admin|program_admin|system`)

**Request**

```json
{
  "terminal_id": "uuid",
  "participant_id": "uuid",
  "program_id": "uuid",
  "location_id": "uuid",
  "client_payment_ref": "offline-001",
  "amount_cents": 1900,
  "currency": "POINTS",
  "occurred_at": "2026-02-21T20:12:00Z",
  "qr_token": "stqr_...",
  "external_order_ref": "order-123",
  "metadata": {
    "offline_reason": "network_outage"
  }
}
```

**Rules**

* Single payment per request (batch sync is deferred).
* Replay identity requires payload hash parity.
* Terminal dedupe identity: `(tenant_id, merchant_id, terminal_id, client_payment_ref)`.
* Replay mismatch on idempotency or terminal dedupe identity returns `409 idempotency_conflict`.

**Response**

```json
{
  "status": "ok",
  "data": {
    "idempotency": "new",
    "offline_payment_id": "uuid",
    "merchant_id": "uuid",
    "terminal_id": "uuid",
    "participant_id": "uuid",
    "client_payment_ref": "offline-001",
    "amount_cents": 1900,
    "currency": "POINTS",
    "status": "synced",
    "attempt_count": 1,
    "failure_reason": null,
    "redemption_id": "uuid"
  }
}
```

**Status lifecycle**

* `pending` -> transient failure; retry metadata is set (`attempt_count`, `next_retry_at`).
* `synced` -> redemption initiated/authorized and linked.
* `failed` -> terminal failure reason (for example `insufficient_balance`) or timeout (`offline_sync_timeout`).

**Side effects**

* First write: `merchant.offline_payment_recorded`
* Success finalize: `merchant.offline_payment_synced`
* Failure finalize: `merchant.offline_payment_failed`

### POST /merchants/:id/pos-events

* `X-Actor-Id: <uuid>` (optional)
* `X-Actor-Role: <role>` (optional; write roles: `tenant_admin|program_admin|system`)

**Request**

```json
{
  "provider": "square",
  "event_type": "sale_completed",
  "event_occurred_at": "2026-02-17T21:10:00Z",
  "dedupe_key": "square:store-12:txn-991",
  "provider_event_id": "evt_991",
  "amount_cents": 1900,
  "currency": "USD",
  "external_order_ref": "order-991",
  "external_payment_ref": "payment-991",
  "raw_payload": {
    "provider_event_id": "evt_991",
    "line_items": [{"sku": "coffee", "qty": 1}]
  },
  "normalized_data": {
    "channel": "pos"
  },
  "metadata": {
    "source": "terminal-7"
  }
}
```

**Response**

```json
{
  "status": "ok",
  "data": {
    "idempotency": "new",
    "pos_event_id": "uuid",
    "merchant_id": "uuid",
    "provider": "square",
    "event_type": "sale_completed",
    "dedupe_key": "square:store-12:txn-991",
```

* Same `(tenant_id, Idempotency-Key)` + same payload identity returns `idempotency: replayed`.
* Same `(tenant_id, merchant_id, dedupe_key)` + same payload identity returns `idempotency: replayed`.

**Side effects (first-time ingest only)**

* Audit log action: `merchant.pos_event_ingested`

### POST /integrations/:id/merchants/:merchant_id/pos-webhook-events

Ingest a provider callback, validate signature when configured, normalize through provider adapters, and persist deterministic webhook receipt/process state.

**Headers**

* `X-Tenant-Id: <tenant_id>` (required)
* `X-Provider-Signature: <signature>` (optional; required when integration metadata includes `webhook_secret`)
* `Idempotency-Key` not required (dedupe uses provider event identity + payload hash)

**Request**

```json
{
  "provider": "square",
  "event_type": "payment.completed",
  "provider_event_id": "square:webhook:evt-001",
  "dedupe_key": "square:webhook:dedupe-001",
  "event_occurred_at": "2026-02-21T18:30:00Z",
  "payload": {
    "id": "payment-001",
    "order_id": "order-001",
    "amount_money": {"amount": 1900, "currency": "USD"}
  },
  "metadata": {"source": "square_webhook"}
}
```

**Response**

```json
{
  "status": "ok",
  "data": {
    "idempotency": "new",
    "pos_webhook_receipt_id": "uuid",
    "integration_id": "uuid",
    "merchant_id": "uuid",
    "provider": "square",
    "event_type": "payment.completed",
    "dedupe_key": "square:webhook:dedupe-001",
    "status": "processed",
    "signature_status": "valid",
    "processing_error": null,
    "linked_pos_event_id": "uuid"
  }
}
```

**Replay/conflict semantics**

* Same `(tenant_id, integration_id, merchant_id, provider, provider_event_id|dedupe_key)` and same payload hash returns deterministic replay.
* Same identity and mismatched payload hash returns `409 idempotency_conflict`.
* Invalid signature returns `403 forbidden` and persists a rejected receipt for controlled reprocess.

**Side effects**

* Audit actions: `merchant.pos_webhook_received`, `merchant.pos_webhook_processed`
* Outbox events: `merchant.pos_webhook_received`, `merchant.pos_webhook_processed`
* Normalized first-time capture still emits canonical `merchant.pos_event_ingested`.

`POST /munus/offerings` is a tenant-scoped write endpoint used by guided custom-offering UI flows.

Frontend behavior note:

* Operator Console guided flow hides route-level endpoint labels in normal mode.
* Endpoint details and raw developer diagnostics are shown only when Munus debug mode is enabled.
* Guided deploy preflight enforces advisor signoff evidence capture via:
* `POST /api/v1/munus/offerings/:id/disclosures`
* `POST /api/v1/munus/offerings/:id/disclosures/:disclosure_id/acknowledgements`
* `POST /api/v1/munus/offerings/:id/filings/preflight`
* `POST /api/v1/munus/offerings/:id/filings`
* `POST /api/v1/munus/filings/:id/publish-confirmations`

Headers:

* `X-Tenant-Id` (required)
* `X-Actor-Role` (optional; write roles: `tenant_admin|program_admin|system`)

Request body:

```json
{
  "offering": {
    "program_id": "uuid",
    "name": "Detroit Revitalization 2026 A",
    "offering_type": "municipal_bond",
    "terms": {
      "series": "Series 2026-A",
      "status": {
        "external": "pre_issuance",
        "mapped_offering_status": "draft"
      },
      "tokenization": {
        "canonical_mode": "hts"
      }
    },
    "metadata": {
      "source": "frontend.operator_console.guided_custom_offering",
      "guided_path": "custom"
    }
  }
}
```

Response `data` includes:

* `offering_id`
* `program_id`
* `name`
* `offering_type`
* `status` (`draft` on create)
* `terms`
* `metadata`

`deploy-token`, `open`, and `close` are money-adjacent writes and require `Idempotency-Key`.

Deterministic replay semantics:

* Same `(tenant_id, idempotency_key, payload_hash)` returns `200` with `data.idempotency = "replayed"`.
* Same `(tenant_id, idempotency_key)` with changed payload returns `409 idempotency_conflict`.
* Missing `Idempotency-Key` returns `400 validation_error`.

`deploy-token` request body:

```json
{
  "deployment": {
    "canonical_mode": "hts",
    "token_class_id": "uuid-or-null",
    "token_name": "Detroit Revitalization 2026 A",
    "token_symbol": "DR26A",
    "decimals": 0,
    "initial_supply": "0",
    "treasury_wallet": "0xabc...",
    "metadata_uri": "ipfs://...",
    "policy_override": {},
    "metadata": {}
  }
}
```

`open`/`close` request body:

```json
{
  "lifecycle": {
    "reason": "issuer_approved_open",
    "canonical_mode": "hts",
    "metadata": {}
  }
}
```

Lifecycle status semantics:

* Initial deploy/open/close writes create token actions in `pending_approvals`.
* Chain submission occurs only after approval threshold, moving token action to `submitted`.
* Offering transitions are finality-driven:
  * `draft -> active` only after confirmed `open`.
  * `active -> closed` only after confirmed `close`.
* `GET /offerings/:id/token-deployments` returns deployment rows with `requested|submitted|confirmed|failed` progression and chain references (`tx_intent_id`, `tx_hash`) when present.

Required metadata for `kms_topic_create`:

* `custody_signer_id`
* `hedera_account_id`
* `hedera_public_key`

Optional fields for `kms_topic_create`:

* `request.topic_memo`
* `metadata.signing_algorithm` (default `ED25519_SHA_512`)
* `metadata.message_type` (default `RAW`)

When the tx-intent finalizes, receipt metadata includes custody/KMS correlation fields:
`sign_job_id`, `aws_request_id`, `kms_key_id`, `signing_algorithm`, `hedera_tx_id`, `receipt_status`.

**Behavior (async finality lifecycle)**

* Default (sync) submit returns `status: "submitted"` and emits `chain.tx_submitted`.
* Feature-flagged async dispatch mode may return `status: "initiated"` with `dispatch_status: "queued"` and emits `chain.tx_queued`.
* Intent lifecycle remains within existing status vocabulary:
  * sync path: `submitted -> confirmed|failed`
  * async path: `initiated -> submitted -> confirmed|failed` (or `failed` on terminal dispatch failure)
* Finality outcomes are emitted asynchronously by the finality worker:
  * `chain.tx_confirmed`
  * `chain.tx_failed`
* Idempotent replay/conflict semantics remain unchanged.

**Aztec runtime scaffold notes**

* `runtime_kind=aztec` resolves to dedicated Aztec runtime module in runtime registry v2.
* Aztec runtime activation is tenant-flagged (`aztec_runtime_enabled`).
* Money-adjacent Aztec tx-intents can require compliance preflight (`aztec_compliance_preflight_enabled`):
  * missing verdict -> `409 validation_error`
  * expired verdict -> `409 validation_error`

**Multichain placeholder scaffold notes**

* Placeholder chain runtimes are available for `base`, `arbitrum`, `ink`, `monad`, `sei`, `solana`, `spark`, `tempo`, `canton`, and `xrpl`.
* `runtime_kind=cari` resolves to a dedicated Cari runtime scaffold and preserves additive DvP metadata passthrough (`dvp_policy_ref`, `dvp_runtime_adapter_ref`) when present.
* These placeholder adapters are deterministic scaffolds and do not perform live network RPC/indexer/prover operations.
